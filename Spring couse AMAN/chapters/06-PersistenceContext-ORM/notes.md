## 1. JpaRepository — Hierarchy and Implementation

When you write `extends JpaRepository<Patient, Long>`, you don't just get a few methods — you get a deep inheritance chain:

```
JpaRepository
  extends PagingAndSortingRepository (adds pagination)
    extends CrudRepository (basic CRUD)
      extends Repository (marker interface)

JpaRepository also extends:
  - ListJpaRepository
  - ListCrudRepository
  - QueryByExampleExecutor
```

All these interfaces have their methods **concretely implemented** in a single class:

**`SimpleJpaRepository<T, ID>`** — this is the real class generated by Spring Data JPA at runtime. It:
- Gets injected with an `EntityManager`
- Implements `findAll()`, `findById()`, `save()`, `delete()`, `existsById()`, `count()`, and many more
- All of these ultimately call `EntityManager` methods internally

## 2. EntityManager — The Core JPA Interface

`EntityManager` is the central JPA interface. It is what actually talks to the database through Hibernate and JDBC.

### Key EntityManager methods

| Method | What it does |
|---|---|
| `em.persist(entity)` | INSERT — adds a new entity to the persistence context |
| `em.find(Class, id)` | SELECT by PK — retrieves entity (checks cache first) |
| `em.merge(entity)` | UPDATE — merges a detached entity back into context |
| `em.remove(entity)` | DELETE — marks entity for removal |
| `em.detach(entity)` | Detaches entity from the context |
| `em.createQuery(jpql)` | Creates a JPQL query |
| `em.flush()` | Forces synchronisation with DB immediately |
| `em.contains(entity)` | Checks if entity is in the persistence context |

> You can inject EntityManager directly, but since JpaRepository is a higher-level abstraction, most developers use that instead.

```java
// Injecting EntityManager directly (advanced use case)
@PersistenceContext
private EntityManager em;
```

## 3. Hibernate Entity Lifecycle

Every JPA entity passes through distinct **lifecycle states**. Understanding these states explains how Hibernate decides when to run SQL queries.

```
  new Object()
       |
       v
  [TRANSIENT]  ← Object exists in Java memory only; NOT tracked by JPA
       |
   persist() / save()
       |
       v
  [PERSISTENT] ← Object is in PersistenceContext (first-level cache); 
       |          DB operations queued until transaction commits
  detach()  |  remove()
       |         |
       v         v
  [DETACHED]  [REMOVED]
       |              |
  merge()/save()      |
       |         garbage collected
  [PERSISTENT]        on commit
```

### 3.1 Transient State

```java
Patient p = new Patient();  // Transient — no DB tracking, no persistence context
p.setName("Anuj");
// If you never call save()/persist(), this object is garbage collected
```

### 3.2 Persistent State

```java
// After save() or findById() — entity is in the Persistence Context
Patient saved = patientRepository.save(p);  // → Persistent state
Patient found = patientRepository.findById(1L).orElseThrow();  // → Persistent state
```

- Object is tracked by the EntityManager
- Any changes are automatically detected (**dirty checking**) at transaction commit

### 3.3 Detached State

```java
// After the transaction/session ends, entity becomes Detached
// Changes to detached entities are NOT saved to DB automatically
// You can re-attach using merge()
patientRepository.save(detachedPatient);  // merges back to Persistent
```

### 3.4 Removed State

```java
patientRepository.deleteById(id);  // entity moves to Removed state
// Will be garbage collected after commit; cannot be persisted again
```

## 4. PersistenceContext — The First-Level Cache

The **PersistenceContext** is an in-memory cache associated with a single transaction/session.

### Key behaviours:

1. **Cache hit optimisation:** If you call `findById(1L)` twice within the same transaction, the second call returns the already-cached object without hitting the DB.

```java
@Transactional
public Patient getPatient(Long id) {
    Patient p1 = patientRepository.findById(id).orElseThrow(); // → DB query
    Patient p2 = patientRepository.findById(id).orElseThrow(); // → Cache hit (no DB)
    System.out.println(p1 == p2);  // → true (same object!)
    return p1;
}
```

Without `@Transactional`:
- Each call to `findById` creates its own transaction → 2 separate DB queries
- `p1 == p2` would be `false` (different objects)

With `@Transactional`:
- One shared transaction → `p1` is cached → `p2` hits cache → `p1 == p2` is `true`

2. **Life span:** PersistenceContext lives for the duration of one transaction/session.

3. **Scope:** One PersistenceContext per HTTP request (in web apps using Spring's `@Transactional`).

## 5. @Transactional — Transaction Management

```java
import jakarta.persistence.Transactional;

@Service
public class PatientService {

    @Transactional   // All operations here share one transaction and one PersistenceContext
    public Patient getPatientById(Long id) {
        Patient p1 = patientRepository.findById(id).orElseThrow();
        Patient p2 = patientRepository.findById(id).orElseThrow();
        // p1 == p2 → true (same object from persistence context)
        return p1;
    }
}
```

**What @Transactional guarantees:**
- All DB operations within the method are part of one transaction
- If the method completes **without error** → transaction is **committed**
- If an **unchecked exception** is thrown → transaction is **rolled back**
- All changes are atomic — either all happen or none

> Without `@Transactional`, each `repository.save()` / `findById()` etc. creates its own tiny transaction.

## 6. Dirty Checking — Automatic Updates

When an entity is in the **Persistent state** within a `@Transactional` context, Hibernate **automatically tracks all field changes**. At commit time, it checks if anything changed (dirty check) and generates UPDATE queries.

```java
@Transactional
public Patient updateName(Long id, String newName) {
    Patient patient = patientRepository.findById(id).orElseThrow();
    // patient is now PERSISTENT
    patient.setName(newName);   // ← No explicit save() needed!
    // At commit: Hibernate detects name changed → auto-generates UPDATE query
    return patient;
}
```

**Hibernate generates:**
```sql
UPDATE patient SET name='Anuj' WHERE id=1
```
**Without you calling `save()` again!**

This is **dirty checking** — Hibernate compares the snapshot taken when the entity was loaded vs its current state at commit time.

## 7. @Table Annotation — Table Configuration

```java
@Entity
@Table(
    name = "patient_tbl",           // custom table name
    schema = "hr",                  // put in a specific schema
    uniqueConstraints = {
        @UniqueConstraint(
            name = "unique_patient_email",
            columnNames = "email"
        ),
        @UniqueConstraint(
            name = "unique_patient_name_birthdate",
            columnNames = {"name", "birth_date"}  // composite unique
        )
    },
    indexes = {
        @Index(
            name = "idx_patient_birth_date",
            columnList = "birth_date"
        )
    }
)
public class Patient { ... }
```

### 7.1 Naming Convention

| Java class name | Default table name | After @Table(name=...) |
|---|---|---|
| `Patient` | `patient` | Whatever you specify |
| `PatientTable` | `patient_table` (snake_case) | Whatever you specify |

Spring Boot uses **snake_case** conversion by default for multi-word class names.

## 8. @UniqueConstraint — Enforcing Uniqueness

Two approaches:

**Option 1: Single column — use `@Column(unique = true)`**
```java
@Column(unique = true)
private String email;
```

**Option 2: Multi-column or named — use `@UniqueConstraint` in `@Table`**
```java
@UniqueConstraint(
    name = "unique_name_birthdate",
    columnNames = {"name", "birth_date"}
)
```

> When a unique constraint is created, Hibernate also **automatically creates a DB index** for it (queries on unique columns are faster).

---

## 9. @Index — Database Indexes

Indexes speed up **SELECT/WHERE** queries by reducing full table scans. Trade-off: they slow down INSERT/UPDATE.

```java
@Index(name = "idx_patient_birth_date", columnList = "birth_date")

// Multiple columns:
@Index(name = "idx_patient_email_name", columnList = "email, name")
```

**Analogy:** Index in a book. Instead of reading every page, you find the page number in the index and jump directly.

| Operation | Without Index | With Index |
|---|---|---|
| SELECT WHERE | O(n) — scan all rows | O(log n) — use B-tree |
| INSERT | Fast | Slower (index must be updated) |
| Storage | Less | More |

> Create indexes on columns you frequently filter by (WHERE, ORDER BY, JOIN ON).

## 10. @Column Annotation — Column Configuration

```java
@Column(
    name = "patient_name",    // custom DB column name
    unique = true,             // adds unique constraint
    nullable = false,          // NOT NULL constraint
    updatable = false,         // column can never be updated after insert
    insertable = true,         // default: column is included in INSERT
    length = 40                // VARCHAR(40) instead of default VARCHAR(255)
)
private String name;
```

### Common @Column properties

| Property | Default | Description |
|---|---|---|
| `name` | field name (snake_case) | Column name in DB |
| `nullable` | `true` | If `false`, adds NOT NULL constraint |
| `unique` | `false` | If `true`, adds unique constraint |
| `length` | `255` | For VARCHAR columns |
| `precision` | `0` | Total digits for DECIMAL |
| `scale` | `0` | Digits after decimal point |
| `insertable` | `true` | Include in INSERT statements |
| `updatable` | `true` | Include in UPDATE statements |

## 11. @CreationTimestamp — Auto Timestamp

```java
import org.hibernate.annotations.CreationTimestamp;

@CreationTimestamp                     // Hibernate annotation — auto-fills on INSERT
@Column(updatable = false)             // prevents accidental update
private LocalDateTime createdAt;
```

- Set automatically when the entity is first persisted
- Not updated on subsequent saves
- Use `@UpdateTimestamp` for the last-modified timestamp

> Note: `@CreationTimestamp` is a **Hibernate** annotation (not JPA standard). It is in `org.hibernate.annotations`.

## 12. @Enumerated — Storing Enums in DB

```java
// Enum definition
public enum BloodGroup {
    A_POSITIVE, A_NEGATIVE,
    B_POSITIVE, B_NEGATIVE,
    AB_POSITIVE, AB_NEGATIVE,
    O_POSITIVE, O_NEGATIVE
}

// In Entity
@Enumerated(EnumType.STRING)   // stores "A_POSITIVE" instead of 0
private BloodGroup bloodGroup;
```

### EnumType.ORDINAL vs EnumType.STRING

| Type | Stored in DB | Pros | Cons |
|---|---|---|---|
| `ORDINAL` | `0`, `1`, `2`... | Less storage | Fragile — reordering enum breaks data |
| `STRING` | `"A_POSITIVE"` | Readable, stable | More storage |

> **Best practice:** Use `EnumType.STRING` for new projects. `ORDINAL` saves space but is risky if you add/reorder enum values.

When `@Enumerated(EnumType.STRING)` is used, Hibernate also generates a **DB-level CHECK constraint** to ensure only valid enum values are stored.

## 13. Data Seeding with data.sql

To pre-populate the DB with test data on startup:

**Step 1 — Create `src/main/resources/data.sql`:**
```sql
INSERT INTO patient (name, gender, birth_date, email, blood_group)
VALUES ('Anuj Sharma', 'MALE', '1995-08-27', 'anuj@example.com', 'O_POSITIVE');

INSERT INTO patient (name, gender, birth_date, email, blood_group)
VALUES ('Priya Patel', 'FEMALE', '1998-03-15', 'priya@example.com', 'A_POSITIVE');
```

**Step 2 — Configure `application.properties`:**
```properties
spring.jpa.defer-datasource-initialization=true
spring.sql.init.mode=always
spring.sql.init.data-locations=classpath:data.sql
```

> `defer-datasource-initialization=true` is needed so that Hibernate creates the tables FIRST, then data.sql runs. Without this, the INSERT fails because the table doesn't exist yet.

## 14. Column Rename — Why Migrations Are Needed

**Critical warning about `ddl-auto=update`:**

If you rename a field in your Entity (e.g., `name` → `patientName`), Hibernate does NOT rename the column. It creates a **new column** `patient_name` while the old `name` column stays.

- Old records: `name` = "Anuj", `patient_name` = NULL
- New records: `name` = NULL, `patient_name` = "Anuj"

**This corrupts your data.**

**Solution for production:** Use **Flyway** or **Liquibase** migration tools to write explicit `ALTER TABLE` scripts.

## 15. Summary — Key Concepts

| Concept | Description |
|---|---|
| `SimpleJpaRepository` | Spring-generated class that implements all JpaRepository methods using EntityManager |
| `EntityManager` | Core JPA interface; all DB operations (persist, find, remove, merge) |
| Transient state | New Java object; not tracked by JPA |
| Persistent state | Entity is in PersistenceContext; changes auto-tracked |
| Detached state | Entity disconnected from context; changes not saved |
| Removed state | Entity marked for deletion; garbage collected after commit |
| PersistenceContext | First-level cache per transaction; avoids duplicate DB calls |
| `@Transactional` | Marks a method as one DB transaction; enables dirty checking + cache |
| Dirty Checking | Hibernate auto-detects field changes in Persistent entities → generates UPDATE |
| `@Table` | Configures table name, schema, unique constraints, indexes |
| `@UniqueConstraint` | Enforces column uniqueness at DB level (single or multi-column) |
| `@Index` | Creates a DB index for faster SELECT queries |
| `@Column` | Configures individual column properties (name, nullable, unique, length) |
| `@CreationTimestamp` | Hibernate: auto-fills timestamp when entity is first persisted |
| `@Enumerated(STRING)` | Stores enum name (not ordinal) in DB; includes DB CHECK constraint |
| data.sql | SQL file auto-executed on startup to seed data |
| Flyway / Liquibase | Migration tools for safe production schema changes |
