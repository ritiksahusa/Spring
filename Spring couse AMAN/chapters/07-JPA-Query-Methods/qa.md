# Chapter 07 — QA: JPA Query Methods, JPQL, Native Queries, Projection, Pagination

---

## Part A — Short Answer Questions (20)

**A1.** What are JPA Derived Query Methods? How does Spring generate SQL from them?

> **Answer:** Derived query methods are repository interface methods whose SQL is auto-generated by Spring Data JPA by parsing the method name. For example, `findByName(String name)` generates `SELECT ... FROM patient WHERE name = ?`. Spring validates field existence at startup and throws an error if invalid.

---

**A2.** What happens if you write a derived query method with a field name that doesn't exist in the entity?

> **Answer:** Spring Boot fails at startup with an error like `No property 'xyz' found for type 'Patient'`. This is fail-fast behavior — bugs are caught at startup, not at runtime.

---

**A3.** What is the difference between `findByNameContaining` and `findByNameLike`?

> **Answer:** `findByNameContaining("anuj")` wraps the value with `%` automatically → `WHERE name LIKE '%anuj%'`. `findByNameLike` does NOT add `%` automatically — you must pass the full pattern yourself: `findByNameLike("%anuj%")`.

---

**A4.** What keyword do you add to a derived query method to sort results?

> **Answer:** `OrderBy` followed by the field name and optionally `Asc` or `Desc`. Example: `findByNameContainingOrderByIdDesc(String query)` → `ORDER BY id DESC`.

---

**A5.** What is JPQL? How is it different from SQL?

> **Answer:** JPQL (Jakarta Persistence Query Language) is object-oriented query language that operates on entity class names and Java field names, not on table names and column names. It is database-independent — Hibernate translates JPQL to the appropriate SQL dialect.

---

**A6.** Write a JPQL `@Query` that finds all patients born after a given date.

> **Answer:**
> ```java
> @Query("SELECT p FROM Patient p WHERE p.birthDate > :birthDate")
> List<Patient> findByBornAfterDate(@Param("birthDate") LocalDate birthDate);
> ```

---

**A7.** What is the difference between named parameters (`:paramName`) and positional parameters (`?1`) in JPQL?

> **Answer:** Named parameters use `:name` and require `@Param("name")` on the method argument. Positional parameters use `?1`, `?2` by position order. Named parameters are preferred for readability, especially in complex queries.

---

**A8.** When do you need `@Modifying` on a `@Query` method? What does it do?

> **Answer:** `@Modifying` is required for UPDATE and DELETE JPQL queries. It tells Spring Data JPA that the query will modify data (not just read it). Without it, executing an UPDATE/DELETE throws `InvalidDataAccessApiUsageException`.

---

**A9.** Does `@Modifying` alone suffice for an update query? What else is required?

> **Answer:** No. `@Transactional` is also required on the repository method (or the calling service method). `@Modifying` marks the intent; `@Transactional` provides the transaction context for the write operation.

---

**A10.** What does `nativeQuery = true` in `@Query` do?

> **Answer:** It sends the query string directly to the JDBC driver as raw SQL, bypassing Hibernate's JPQL parsing and dialect translation. You must use actual table/column names, not entity/field names.

---

**A11.** Can you use JPQL projection (`NEW ClassName(...)`) with `nativeQuery = true`? Why or why not?

> **Answer:** No. The `NEW` constructor expression is a JPQL/Hibernate feature. With `nativeQuery = true`, Hibernate doesn't process the results, so it cannot call the DTO constructor. You must use `List<Object[]>` or interface projections with native queries.

---

**A12.** What is a Projection DTO in JPQL? What annotation must it have?

> **Answer:** A Projection DTO is a class that holds a subset of query results. In JPQL, it is instantiated via `SELECT NEW fully.qualified.DtoClass(field1, field2)`. The DTO **must** have `@AllArgsConstructor` (or a matching constructor) because Hibernate calls the constructor to build each result object.

---

**A13.** What is `Pageable` in Spring Data JPA?

> **Answer:** `Pageable` is an interface that carries pagination information: page number, page size, and optional sorting. It is passed to repository methods to limit query results. `PageRequest.of(page, size)` creates a concrete `Pageable` instance.

---

**A14.** Are page numbers in `PageRequest.of()` 0-indexed or 1-indexed?

> **Answer:** 0-indexed. `PageRequest.of(0, 10)` returns the first 10 records (page 1 from the user's perspective).

---

**A15.** What does `Page<T>` provide over a plain `List<T>`?

> **Answer:** `Page<T>` provides both the result data (`getContent()`) and pagination metadata: `getTotalPages()`, `getTotalElements()`, `hasNext()`, `hasPrevious()`, `getNumber()`, `getSize()`. This is ideal for REST APIs where the client needs metadata to render pagination controls.

---

**A16.** How many SQL queries does Spring execute when you call a paginated `findAll(pageable)`?

> **Answer:** Two SQL queries: (1) a `SELECT ... LIMIT x OFFSET y` to fetch the current page's records, and (2) a `SELECT COUNT(1)` to get the total record count for computing `totalPages` and `totalElements`.

---

**A17.** What does a `GROUP BY` query in JPQL return when not using projection?

> **Answer:** `List<Object[]>` — each element is an array where each position corresponds to one selected field in the query. For example, `SELECT p.bloodGroup, COUNT(p) GROUP BY ...` returns `Object[]{BloodGroupType, Long}` for each row.

---

**A18.** What is the difference between `findByName` and `findByNameContaining`?

> **Answer:** `findByName(String name)` generates `WHERE name = ?` (exact match). `findByNameContaining(String query)` generates `WHERE name LIKE '%?%'` (substring match).

---

**A19.** How do you add sorting to a `PageRequest`?

> **Answer:** Use the overloaded `PageRequest.of(page, size, Sort.by("fieldName"))`. For descending: `Sort.by("fieldName").descending()`. For multiple fields: `Sort.by("name", "birthDate")`.

---

**A20.** Write the fully qualified JPQL projection query for counting patients per blood group, mapping to `BloodGroupCountDto`.

> **Answer:**
> ```java
> @Query("SELECT NEW com.example.dto.BloodGroupCountDto(p.bloodGroup, COUNT(p)) " +
>        "FROM Patient p GROUP BY p.bloodGroup")
> List<BloodGroupCountDto> countEachBloodGroupType();
> ```

**C3.** Write the repository and service code to update a patient's phone number given their ID.

> **Answer:**
> ```java
> // Repository
> @Modifying
> @Transactional
> @Query("UPDATE Patient p SET p.phone = :phone WHERE p.id = :id")
> int updatePhoneById(@Param("phone") String phone, @Param("id") Long id);
>
> // Service
> public void updatePhone(Long id, String phone) {
>     int rows = patientRepository.updatePhoneById(phone, id);
>     if (rows == 0) throw new ResourceNotFoundException("Patient not found");
> }
> ```

---

**C4.** A junior developer writes this native query:
> `@Query(value = "SELECT NEW dto.PatientSummaryDto(p.name, p.email) FROM patient p", nativeQuery = true)`
>
> What is wrong with it?

> **Answer:** Two problems:
> 1. `NEW constructor expression` doesn't work with `nativeQuery = true` — Hibernate doesn't process native query results.
> 2. In native queries, you reference table/column names (`patient`), not entity names — but the syntax with `NEW` is JPQL-only.
> Fix: Remove `nativeQuery = true` and use JPQL with the entity name `Patient`, OR keep native query and return `List<Object[]>`.

---

**C5.** Write a paginated endpoint that returns page 0 with 5 patients sorted by name.

> **Answer:**
> ```java
> // Repository
> Page<Patient> findAll(Pageable pageable);
>
> // Controller
> @GetMapping("/patients")
> public ResponseEntity<Page<Patient>> getAll(
>     @RequestParam(defaultValue = "0") int page,
>     @RequestParam(defaultValue = "5") int size) {
>
>     Pageable pageable = PageRequest.of(page, size, Sort.by("name"));
>     return ResponseEntity.ok(patientRepository.findAll(pageable));
> }
> ```

## Part F — 🚀 Advanced / Real-World Interview Questions (Product-Company Level)

> These go beyond the transcript — the kind of follow-up/curveball questions asked at senior or product-company interviews.

**F1.** Why does deep OFFSET pagination (`PageRequest.of(10000, 20)`) get progressively slower as the page number grows, and what's the standard fix?

> **Answer:** `OFFSET 200000 LIMIT 20` forces the DB to scan and discard 200,000 rows before returning the next 20 — cost grows linearly with offset. The fix is **keyset/cursor (seek) pagination**: filter `WHERE id > :lastSeenId ORDER BY id LIMIT 20` on an indexed column instead of using `OFFSET`, so the DB jumps directly to the right spot regardless of depth.

**F2.** What's the difference between `Page<T>` and `Slice<T>`, and when would you prefer `Slice`?

> **Answer:** `Page<T>` runs an extra `COUNT(*)` query for `getTotalElements()`/`getTotalPages()`, which can be expensive on large tables. `Slice<T>` skips the count query and only exposes `hasNext()` (by fetching `pageSize + 1` rows internally). Prefer `Slice` for infinite-scroll UIs that don't need an exact total.

**F3.** How would you implement a dynamic search endpoint with many optional filters WITHOUT writing a combinatorial explosion of derived query methods?

> **Answer:** Use the Specification API (`JpaSpecificationExecutor<T>`), composing `Predicate`s only for non-null filters (`Specification.where(...).and(...)`), or Querydsl for a type-safe fluent builder — both compose WHERE clauses dynamically at runtime instead of hard-coding every filter permutation as a separate `findByXAndY...` method.

**F4.** Your `@Query` combines `JOIN FETCH` with `Pageable`. What warning/problem occurs, and why?

> **Answer:** Hibernate can't apply SQL-level `LIMIT`/`OFFSET` correctly when a `@OneToMany` collection is eagerly joined in the same query (the join multiplies rows). Hibernate either warns and paginates IN MEMORY (loading the entire result set before slicing — risking OOM) or throws outright. Fix: paginate a query returning only parent IDs, then run a SEPARATE `JOIN FETCH ... WHERE id IN (:ids)` query to enrich just that page (the "two-step" pattern).

**F5.** What's the difference between a JPQL/interface-based Projection and `@EntityGraph`?

> **Answer:** A Projection changes WHAT COLUMNS are selected (narrower `SELECT`, less data transfer). `@EntityGraph` changes HOW ASSOCIATIONS are fetched for a specific query (turning a normally-LAZY relation into join-fetched, without altering the entity's default `FetchType`) but still returns the FULL entity shape. Use Projections for summarized/read-only views; use `@EntityGraph` when you need the full entity graph with per-query N+1 control.

**F6.** At what point should a team migrate away from derived query methods, and to what alternatives?

> **Answer:** When method names get long/fragile (multiple `And`/`Or` conditions, hard to read or refactor safely) or logic needs slight variations across call sites. Migrate to `@Query` with explicit JPQL for clarity, the Specification API for truly dynamic/optional filters, or Querydsl for compile-time-safe complex queries.

**F7.** Why might you need an explicit `countQuery` alongside `@Query` + `Pageable` instead of letting Spring Data derive it automatically?

> **Answer:** For complex JPQL (GROUP BY, multiple joins, DISTINCT), Spring Data's auto-derived count query (naively wrapping your SELECT) can be inefficient or semantically wrong. An explicit, simpler `countQuery` (e.g., `SELECT COUNT(p.id) FROM Patient p WHERE ...`) directly improves `Page<T>`'s total-count performance.

**F8.** How would you PROVE, in an automated test, that a repository method issues exactly one SQL query — before merging a PR?

> **Answer:** Use a SQL-statement-counting library like `datasource-proxy` or Hypersistence Utils' `SQLStatementCountValidator` inside a `@DataJpaTest`, asserting `assertSelectCount(1)`. This turns "no N+1 regression" into an enforceable automated test assertion rather than a manual code-review checklist item.
