## Part A — Short Answer Questions (20)

<a id="qA1"></a>**A1.** What are JPA Derived Query Methods? How does Spring generate SQL from them? [↓ Answer](#aA1)

<a id="aA1"></a>> **Answer:** Derived query methods are repository interface methods whose SQL is auto-generated by Spring Data JPA by parsing the method name. For example, `findByName(String name)` generates `SELECT ... FROM patient WHERE name = ?`. Spring validates field existence at startup and throws an error if invalid. [↑ Question](#qA1)

---

<a id="qA2"></a>**A2.** What happens if you write a derived query method with a field name that doesn't exist in the entity? [↓ Answer](#aA2)

<a id="aA2"></a>> **Answer:** Spring Boot fails at startup with an error like `No property 'xyz' found for type 'Patient'`. This is fail-fast behavior — bugs are caught at startup, not at runtime. [↑ Question](#qA2)

---

<a id="qA3"></a>**A3.** What is the difference between `findByNameContaining` and `findByNameLike`? [↓ Answer](#aA3)

<a id="aA3"></a>> **Answer:** `findByNameContaining("anuj")` wraps the value with `%` automatically → `WHERE name LIKE '%anuj%'`. `findByNameLike` does NOT add `%` automatically — you must pass the full pattern yourself: `findByNameLike("%anuj%")`. [↑ Question](#qA3)

---

<a id="qA4"></a>**A4.** What keyword do you add to a derived query method to sort results? [↓ Answer](#aA4)

<a id="aA4"></a>> **Answer:** `OrderBy` followed by the field name and optionally `Asc` or `Desc`. Example: `findByNameContainingOrderByIdDesc(String query)` → `ORDER BY id DESC`. [↑ Question](#qA4)

---

<a id="qA5"></a>**A5.** What is JPQL? How is it different from SQL? [↓ Answer](#aA5)

<a id="aA5"></a>> **Answer:** JPQL (Jakarta Persistence Query Language) is object-oriented query language that operates on entity class names and Java field names, not on table names and column names. It is database-independent — Hibernate translates JPQL to the appropriate SQL dialect. [↑ Question](#qA5)

---

<a id="qA6"></a>**A6.** Write a JPQL `@Query` that finds all patients born after a given date. [↓ Answer](#aA6)

<a id="aA6"></a>> **Answer:**
> ```java
> @Query("SELECT p FROM Patient p WHERE p.birthDate > :birthDate")
> List<Patient> findByBornAfterDate(@Param("birthDate") LocalDate birthDate);
> ```
>
> [↑ Question](#qA6)

---

<a id="qA7"></a>**A7.** What is the difference between named parameters (`:paramName`) and positional parameters (`?1`) in JPQL? [↓ Answer](#aA7)

<a id="aA7"></a>> **Answer:** Named parameters use `:name` and require `@Param("name")` on the method argument. Positional parameters use `?1`, `?2` by position order. Named parameters are preferred for readability, especially in complex queries. [↑ Question](#qA7)

---

<a id="qA8"></a>**A8.** When do you need `@Modifying` on a `@Query` method? What does it do? [↓ Answer](#aA8)

<a id="aA8"></a>> **Answer:** `@Modifying` is required for UPDATE and DELETE JPQL queries. It tells Spring Data JPA that the query will modify data (not just read it). Without it, executing an UPDATE/DELETE throws `InvalidDataAccessApiUsageException`. [↑ Question](#qA8)

---

<a id="qA9"></a>**A9.** Does `@Modifying` alone suffice for an update query? What else is required? [↓ Answer](#aA9)

<a id="aA9"></a>> **Answer:** No. `@Transactional` is also required on the repository method (or the calling service method). `@Modifying` marks the intent; `@Transactional` provides the transaction context for the write operation. [↑ Question](#qA9)

---

<a id="qA10"></a>**A10.** What does `nativeQuery = true` in `@Query` do? [↓ Answer](#aA10)

<a id="aA10"></a>> **Answer:** It sends the query string directly to the JDBC driver as raw SQL, bypassing Hibernate's JPQL parsing and dialect translation. You must use actual table/column names, not entity/field names. [↑ Question](#qA10)

---

<a id="qA11"></a>**A11.** Can you use JPQL projection (`NEW ClassName(...)`) with `nativeQuery = true`? Why or why not? [↓ Answer](#aA11)

<a id="aA11"></a>> **Answer:** No. The `NEW` constructor expression is a JPQL/Hibernate feature. With `nativeQuery = true`, Hibernate doesn't process the results, so it cannot call the DTO constructor. You must use `List<Object[]>` or interface projections with native queries. [↑ Question](#qA11)

---

<a id="qA12"></a>**A12.** What is a Projection DTO in JPQL? What annotation must it have? [↓ Answer](#aA12)

<a id="aA12"></a>> **Answer:** A Projection DTO is a class that holds a subset of query results. In JPQL, it is instantiated via `SELECT NEW fully.qualified.DtoClass(field1, field2)`. The DTO **must** have `@AllArgsConstructor` (or a matching constructor) because Hibernate calls the constructor to build each result object. [↑ Question](#qA12)

---

<a id="qA13"></a>**A13.** What is `Pageable` in Spring Data JPA? [↓ Answer](#aA13)

<a id="aA13"></a>> **Answer:** `Pageable` is an interface that carries pagination information: page number, page size, and optional sorting. It is passed to repository methods to limit query results. `PageRequest.of(page, size)` creates a concrete `Pageable` instance. [↑ Question](#qA13)

---

<a id="qA14"></a>**A14.** Are page numbers in `PageRequest.of()` 0-indexed or 1-indexed? [↓ Answer](#aA14)

<a id="aA14"></a>> **Answer:** 0-indexed. `PageRequest.of(0, 10)` returns the first 10 records (page 1 from the user's perspective). [↑ Question](#qA14)

---

<a id="qA15"></a>**A15.** What does `Page<T>` provide over a plain `List<T>`? [↓ Answer](#aA15)

<a id="aA15"></a>> **Answer:** `Page<T>` provides both the result data (`getContent()`) and pagination metadata: `getTotalPages()`, `getTotalElements()`, `hasNext()`, `hasPrevious()`, `getNumber()`, `getSize()`. This is ideal for REST APIs where the client needs metadata to render pagination controls. [↑ Question](#qA15)

---

<a id="qA16"></a>**A16.** How many SQL queries does Spring execute when you call a paginated `findAll(pageable)`? [↓ Answer](#aA16)

<a id="aA16"></a>> **Answer:** Two SQL queries: (1) a `SELECT ... LIMIT x OFFSET y` to fetch the current page's records, and (2) a `SELECT COUNT(1)` to get the total record count for computing `totalPages` and `totalElements`. [↑ Question](#qA16)

---

<a id="qA17"></a>**A17.** What does a `GROUP BY` query in JPQL return when not using projection? [↓ Answer](#aA17)

<a id="aA17"></a>> **Answer:** `List<Object[]>` — each element is an array where each position corresponds to one selected field in the query. For example, `SELECT p.bloodGroup, COUNT(p) GROUP BY ...` returns `Object[]{BloodGroupType, Long}` for each row. [↑ Question](#qA17)

---

<a id="qA18"></a>**A18.** What is the difference between `findByName` and `findByNameContaining`? [↓ Answer](#aA18)

<a id="aA18"></a>> **Answer:** `findByName(String name)` generates `WHERE name = ?` (exact match). `findByNameContaining(String query)` generates `WHERE name LIKE '%?%'` (substring match). [↑ Question](#qA18)

---

<a id="qA19"></a>**A19.** How do you add sorting to a `PageRequest`? [↓ Answer](#aA19)

<a id="aA19"></a>> **Answer:** Use the overloaded `PageRequest.of(page, size, Sort.by("fieldName"))`. For descending: `Sort.by("fieldName").descending()`. For multiple fields: `Sort.by("name", "birthDate")`. [↑ Question](#qA19)

---

<a id="qA20"></a>**A20.** Write the fully qualified JPQL projection query for counting patients per blood group, mapping to `BloodGroupCountDto`. [↓ Answer](#aA20)

<a id="aA20"></a>> **Answer:**

> @Query("SELECT NEW com.example.dto.BloodGroupCountDto(p.bloodGroup, COUNT(p)) " +
>        "FROM Patient p GROUP BY p.bloodGroup")
> List<BloodGroupCountDto> countEachBloodGroupType();
> ```
>
> [↑ Question](#qA20)

## Part F — 🚀 Advanced / Real-World Interview Questions (Product-Company Level)

> These go beyond the transcript — the kind of follow-up/curveball questions asked at senior or product-company interviews.

<a id="qf1"></a>**F1.** Why does deep OFFSET pagination (`PageRequest.of(10000, 20)`) get progressively slower as the page number grows, and what's the standard fix? [↓ Answer](#af1)

<a id="af1"></a>> **Answer:** `OFFSET 200000 LIMIT 20` forces the DB to scan and discard 200,000 rows before returning the next 20 — cost grows linearly with offset. The fix is **keyset/cursor (seek) pagination**: filter `WHERE id > :lastSeenId ORDER BY id LIMIT 20` on an indexed column instead of using `OFFSET`, so the DB jumps directly to the right spot regardless of depth. [↑ Question](#qf1)

<a id="qf2"></a>**F2.** What's the difference between `Page<T>` and `Slice<T>`, and when would you prefer `Slice`? [↓ Answer](#af2)

<a id="af2"></a>> **Answer:** `Page<T>` runs an extra `COUNT(*)` query for `getTotalElements()`/`getTotalPages()`, which can be expensive on large tables. `Slice<T>` skips the count query and only exposes `hasNext()` (by fetching `pageSize + 1` rows internally). Prefer `Slice` for infinite-scroll UIs that don't need an exact total. [↑ Question](#qf2)

<a id="qf3"></a>**F3.** How would you implement a dynamic search endpoint with many optional filters WITHOUT writing a combinatorial explosion of derived query methods? [↓ Answer](#af3)

<a id="af3"></a>> **Answer:** Use the Specification API (`JpaSpecificationExecutor<T>`), composing `Predicate`s only for non-null filters (`Specification.where(...).and(...)`), or Querydsl for a type-safe fluent builder — both compose WHERE clauses dynamically at runtime instead of hard-coding every filter permutation as a separate `findByXAndY...` method. [↑ Question](#qf3)

<a id="qf4"></a>**F4.** Your `@Query` combines `JOIN FETCH` with `Pageable`. What warning/problem occurs, and why? [↓ Answer](#af4)

<a id="af4"></a>> **Answer:** Hibernate can't apply SQL-level `LIMIT`/`OFFSET` correctly when a `@OneToMany` collection is eagerly joined in the same query (the join multiplies rows). Hibernate either warns and paginates IN MEMORY (loading the entire result set before slicing — risking OOM) or throws outright. Fix: paginate a query returning only parent IDs, then run a SEPARATE `JOIN FETCH ... WHERE id IN (:ids)` query to enrich just that page (the "two-step" pattern). [↑ Question](#qf4)

<a id="qf5"></a>**F5.** What's the difference between a JPQL/interface-based Projection and `@EntityGraph`? [↓ Answer](#af5)

<a id="af5"></a>> **Answer:** A Projection changes WHAT COLUMNS are selected (narrower `SELECT`, less data transfer). `@EntityGraph` changes HOW ASSOCIATIONS are fetched for a specific query (turning a normally-LAZY relation into join-fetched, without altering the entity's default `FetchType`) but still returns the FULL entity shape. Use Projections for summarized/read-only views; use `@EntityGraph` when you need the full entity graph with per-query N+1 control. [↑ Question](#qf5)

<a id="qf6"></a>**F6.** At what point should a team migrate away from derived query methods, and to what alternatives? [↓ Answer](#af6)

<a id="af6"></a>> **Answer:** When method names get long/fragile (multiple `And`/`Or` conditions, hard to read or refactor safely) or logic needs slight variations across call sites. Migrate to `@Query` with explicit JPQL for clarity, the Specification API for truly dynamic/optional filters, or Querydsl for compile-time-safe complex queries. [↑ Question](#qf6)

<a id="qf7"></a>**F7.** Why might you need an explicit `countQuery` alongside `@Query` + `Pageable` instead of letting Spring Data derive it automatically? [↓ Answer](#af7)

<a id="af7"></a>> **Answer:** For complex JPQL (GROUP BY, multiple joins, DISTINCT), Spring Data's auto-derived count query (naively wrapping your SELECT) can be inefficient or semantically wrong. An explicit, simpler `countQuery` (e.g., `SELECT COUNT(p.id) FROM Patient p WHERE ...`) directly improves `Page<T>`'s total-count performance. [↑ Question](#qf7)

<a id="qf8"></a>**F8.** How would you PROVE, in an automated test, that a repository method issues exactly one SQL query — before merging a PR? [↓ Answer](#af8)

<a id="af8"></a>> **Answer:** Use a SQL-statement-counting library like `datasource-proxy` or Hypersistence Utils' `SQLStatementCountValidator` inside a `@DataJpaTest`, asserting `assertSelectCount(1)`. This turns "no N+1 regression" into an enforceable automated test assertion rather than a manual code-review checklist item. [↑ Question](#qf8)
